Maven中的scope

在maven中，scope用于定义依赖性的作用范围，即指定依赖在项目的哪些阶段（编译、测试、运行）生效，以及是否会被传递到其他依赖中。不同的scope会影响依赖的打包行为、传递性和可用性。

scope 说明 典型场景示例
compile 默认值。依赖在编译、测试、运行时均可用，且会传递到依赖的项目中 Spring Core、Mybatis等
provided 依赖在编译和测试时可用，但运行时由环节提供，不会打包到最终产物 Servlet API
runtime 依赖在测试和运行时可用，但编译时不需要 ​
test 依赖仅在测试阶段可用，不会传递到主代码或打包产品中 Junit、Mockito
system 与provided类似，但依赖需要显式指定本地路径（不推荐使用，破坏可移植性） 本地非Maven仓库的特殊性
import 仅在dependencyManagement中使用，用于导入其他POM的依赖管理配置 ​
关键区别

  1. 依赖传递性：compile、runtime会传递依赖；provided、test、system不会传递依赖

  2. 打包行为：compile、runtime会打包到最终产物（如JAR/WAR）；provided、test、system不会打包到最终产物中

  3. 生命周期阶段：test仅影响测试阶段；provided在运行时不生效（由环境提供）### 什么是Maven仓库

Maven依赖原则

  1. 依赖路径最短优先原则：一个项目依赖了两个jar包，其中A-B-C-X(1.0)，A-D-X(2.0)。由于X(2.0)路径最短，所以项目使用的是X(2.0)
  2. pom文件中申明顺序优先：如果A-B-X(1.0)，A-C-X(2.0)，这样的路径长度一样怎么办？这样的情况下，maven会根据pom文件声明的顺序加载，如果先声明了B，后声明了C，那就最终的依就会是X(1.0)
  3. 覆写优先，子项目中pom内声明的优先于父项目pom中的依赖### 如果解决jar冲突
    遇到冲突的第一步就是 通过mvn dependency:tree 查看依赖树或者idea maven helper插件 找到maven加载的是什么版本的jar包。然后，通过maven的依赖原则来调整坐标在pom文件中的声明顺序是最好的办法，或者将冲突中不想要的jar掉。

Maven的生命周期

Maven由三套相互独立的生命周期，分别为Clean，Default，Site。每个生命周期包含了一些极端，阶段是有顺序的，后面的阶段依赖于前面的阶段。

  • 1、Clean 生命周期：清理项目，包含三个 phase ：
  • pre-clean：执行清理前需要完成的工作。
  • clean：清理上一次构建生成的文件。
  • post-clean：执行清理后需要完成的工作
  • 2、Default 生命周期：构建项目，重要的 phase 如下：
  • validate：验证工程是否正确，所有需要的资源是否可用。
  • compile：编译项目的源代码。
  • test：使用合适的单元测试框架来测试已编译的源代码。这些测试不需要已打包和布署。
  • package：把已编译的代码打包成可发布的格式，比如 jar、war 等。
  • integration-test：如有需要，将包处理和发布到一个能够进行集成测试的环境。
  • verify：运行所有检查，验证包是否有效且达到质量标准。
  • install：把包安装到maven本地仓库，可以被其他工程作为依赖来使用。
  • deploy：在集成或者发布环境下执行，将最终版本的包拷贝到远程的repository，使得其他的开发者或者工程可以共享。
  • 3、Site 生命周期：建立和发布项目站点，phase 如下：
  • pre-site：生成项目站点之前需要完成的工作
  • site：生成项目站点文档
  • post-site：生成项目站点之后需要完成的工作
  • site-deploy：将项目站点发布到服务器在ieda中，maven生命周期由九种操作：
  1. clean，清理自动生成的文件，也就是target目录
  2. validate，严重maven描述文件是否有效
  3. compile，编译Java代码
  4. test，运行测试代码
  5. package，项目打成jar、war包
  6. verify，验证构建包是否有效
  7. install，将构建包安装到本地仓库
  8. deploy，将构建包部署到远端仓库
  9. site，生成项目站点​
