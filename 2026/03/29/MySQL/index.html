一条sql查询语句是怎么执行的

MySQL分为Server层和执行引擎两个部分组成，其中Server层

  1. 第一步是连接上数据库，客户端根据ip地址，端口号，用户名，用户密码连接数据库，认证通过后，连接器会到权限表中查到用户拥有的权限，之后，这个连接里面的权限判断逻辑，都将依赖此时读到的权限。

  2. 连接完成后，select语句就会查询缓存。mysql拿到一个查询请求后，就先到查询缓存看看，之前是不是执行过这条语句，之前执行过的语句及其结果都会以key-value对的形式，被直接缓存到内存中，其中key为执行的语句，value为查询的结果。不过如果表修改频繁的话，缓存的效率并不高，查询缓存的失效非常频繁。mysql8.0中已经将查询缓存功能删除。

  3. 后面是分析器进行分析，分析器会知道你要做什么。分析器先会做词法分析，如通过select这个关键词知道这是个查询语句，然后做语法分析，判断输入的这个sql语句是否满足mysql的语法规范。

  4. 下面是优化器，优化器是知道怎么做，优化器是在表中有多个索引段时候，决定使用哪个所有，或者在一个语句有多个表关联的时候，决定各个表的连接顺序。

  5. 执行器先判断该用户对这个表有没有执行指令的权限，如果有则调用存储引擎的接口。

  6. 如果查询字段没有索引，那么会调用引擎接口取这个表的第一行，判断是否匹配，如果匹配则将这一行存入结果集中，然后调用引擎接口取下一行，重复相同的判断逻辑，直到取到这个表的最后一行。执行器将上述遍历过程中所有满足条件的行组成的记录集作为结果集返回给而客户端。

  7. 如果查询字段有索引，第一次调用的是“取满足条件的第一行”这个接口，之后循环取“满足条件的下一行”这个接口，这些接口都是引擎中已经定义好的。### 一条SQL更新语句是如何执行的

  8. 首先需要先连接数据库，这是连接器的工作

  9. 在更新一个表的时候，跟这个表有关的查询缓存会失效，所以这条语句就会把表上的所有缓存结果都清空。

  10. 分析器和优化器起到 词法分析，语法分析，优化执行过程的作用

  11. 为了避免每次的更新操作都需要写进磁盘，然后磁盘也要找到对应的那条记录，然后再更新，这个过程id成本、查找成本都很高，MySQL使用到WAL技术，即Write-Aware-Logging，先写日志，再写磁盘。具体来说，当有一条记录需要更新的时候，InnoDB引擎就会先把记录写到redo log里面，并更新内存，这个时候更新就算完成了。同时，InnoDB会在适当的时候，将这个操作记录更新到磁盘中，这个适当的时候是指在系统比较空闲的时候。

  12. 执行器和InnoDB引擎在执行update语句时的内部流程：

  13. 执行器先到引擎中取id=2这一行，id是主键，引擎直接用树搜索找到这一行，如果id=2这一行所在的数据页本来就在内存中，就直接返回给执行器，否则，需要先从磁盘读入内存，然后再返回。

  14. 执行器拿到引擎给的行数据，把这个值加上1，比如原来是N，现在就是N+1，得到新的一行数据，再调用引擎接口写入这行新数据。

  15. 引擎将这行新数据更新到内存中，同时这个更新操作记录到redo log里面，此时redo log处于prepare状态，然后告知执行器执行完成，随时可以提交事务

  16. 执行器生成这个操作的binlog，并把binlog写入磁盘

  17. 执行器调用引擎的提交事务接口，引擎把刚刚写入的redo log改成提交状态，更新完成### 两阶段提交
    两阶段提交是指将redo log的写入拆成两个步骤prepare和commit。这样做的目的是让两份日志之间的逻辑一致。

为什么需要两阶段提交：简单来说，如果不使用两阶段提交，那么很可能导致redo log和bin log之间不一致

  1. 先写 redo log 后写 binlog。假设在 redo log 写完，binlog 还没有写完的时候，MySQL 进程异常重启。由于我们前面说过的，redo log 写完之后，系统即使崩溃，仍然能够把数据恢复回来，所以恢复后这一行 c 的值是 1。 但是由于 binlog 没写完就 crash 了，这时候 binlog 里面就没有记录这个语句。因此，之后备份日志的时候，存起来的 binlog 里面就没有这条语句。 然后你会发现，如果需要用这个 binlog 来恢复临时库的话，由于这个语句的 binlog 丢失，这个临时库就会少了这一次更新，恢复出来的这一行 c 的值就是 0，与原库的值不同。
  2. 先写 binlog 后写 redo log。如果在 binlog 写完之后 crash，由于 redo log 还没写，崩溃恢复以后这个事务无效，所以这一行 c 的值是 0。但是 binlog 里面已经记录了“把 c 从 0 改成 1”这个日志。所以，在之后用 binlog 来恢复的时候就多了一个事务出来，恢复出来的这一行 c 的值就是 1，与原库的值不同### char和varchar的区别
    char的长度是固定长度，如果一个字段为char(10)，那么即使存入的数据长度小于10个字节，那么它占用的空间依然是10个字节，优点是效率高，缺点是占用空间，适用于存储长度固定的数据
    varchar的长度是可变的，存储所占用的空间为数据本身所占用的空间与记录数据占用空间这个值所占用的空间之和。
    从空间上考虑varchar比较合适，从效率上考虑char比较合适，两者使用需要权衡。

什么是索引

索引是出现时为了提高数据的查询效率，就像书的目录一样，我们可以通过目录很快的查到一个章节在哪里。索引的优点是可以加快查询的速度，缺点是索引是占用空间的，创建索引和维护索引是需要耗费时间的，这个时间随着数据量的增加而增加
索引包含着数据表中所有记录的引用指针，使用索引可以提高数据库查询、更新数据的的效率，在MyISAM中索引是一个文件，在InnoDB中索引和数据在同一个文件中

建立索引的几个原则

  1. 最左匹配原则，这是个非常重要的原则，mysql会一直向右匹配直到遇到范围查询（<>，between，like）就停止匹配，比如where子句中为a=1,b=2,c>3,d=4，如果建立（a,b,c,d）顺序的索引，那么d是用不到索引的，如果建立(a,b,d,c)顺序的索引，abd三个顺序随意，因为优化器会将where子句中abd的顺序进行优化至索引中字段的顺序
  2. 尽量选用区分度高的列作为索引，比如性别一般不会在用户表中所谓索引
  3. 索引列不能参与计算，b+树中存的都是数据表中的字段值，但进行索引时，需要把所有元素都应用到函数中才能比较，显然成本太大
  4. 尽量去扩展索引，不要新建索引，如果表中有a这个索引，现在需要(a,b)这个索引，那么只需要修改原来的索引即可。
  5. 为经常需要排序、分组和联合操作的字段建立索引，为常作为查询条件的字段建立索引。
  6. 尽量使用占用空间少的字段作为索引，如果字段的长度长，则可以使用前缀来作为索引。### 为什么索引底层使用B+树，为什么不适用hash表、有序数组
    如果使用hash表需要比较好的hash算法，来解决hash碰撞，数据散列不均匀的问题，而且若使用hash表，在范围查询时，需要遍历查询，效率低下
    有序数组在等值查询和范围查询场景中的性能都十分优秀，但是如果向有序数组中插入一个值，那么就要移动其后的所有值，移动的成本很高，所以有序数组序列只适用于静态存储引擎。
    不适用二叉树类型的树是因为二叉树比B+树的高度要高，查找的效率低，会增加io的效率低，平衡二叉树是通过旋转来保持平衡的，而旋转是对整棵树的操作，若部分加载到内存中则无法完成旋转操作
    B+树比B树的改进的地方是，在B+树中只有在叶子节点中存储数据，分支节点只有key，这样按索引查找的时候，一个磁盘块中能够存储更多的数据，一般情况下3到4层的B+树能够满足千万级的数据，第二个改进的地方是B+树的叶子节点之间式链式结构

对索引B+树的理解

B+树是基于B树和叶子结点顺序访问指针进行实现的，它具有b树的平衡性，并且通过顺序访问指针来提高区间查询的性能。
在b+树中，分支结点中只存在key和指向下一层结点的指针，分支结点中的key从左到右非递减排列，如果某个指针的左右相邻key分别为keyi 和keyi+1，那么指针所指向的结点中所有的key大于等于keyi且小于等于keyi+1。在根据索引查找数据的时候，会先从根节点开始进行二分查找，找到key所在的指针，然后递归的在指针所指向的结点进行查找。直到查找到叶子结点，然后在叶子结点上进行二分查找，找到key所对应的data。
插入删除操作会破坏树的平衡性，因此在插入删除操作后，需要对树进行一个分裂、合并、旋转等操作来维护平衡性。

B+树与B树的区别

  1. B+树非叶子节点不存储数据，所有data存储在叶子节点，导致查询时间复杂度固定为logn，而b树查询时间复杂度不固定，与key在树中的位置有关，最好为O(1)

  2. 由于b+树内节点无data域，每个节点能索引的范围更大更精确

  3. b+树叶子节点两两相连，大大增加了区间访问性，可使用在范围查询等#### B树有如下的特点：

  4. 索引值和具体的data都存在于每个节点中，分布在整棵树中

  5. 每个关键字都会出现，但只出现一次

  6. 搜索有可能在非叶子节点结束

自增主键

插入新纪录的时候可以不指定ID的值，系统会获取当前ID最大值加1作为下一条记录的ID值。也就是说，自增主键的插入数据模式，正符合了递增插入的场景，每次插入一条新纪录，都是追加操作，都不涉及到挪动其他记录，也不会触发叶子节点的分裂。
有业务逻辑的字段做主键，往往不容易保证有序插入，写数据成本相对较高。且业务逻辑的字段长度较大，需要的存储空间较大，如果使用其为主键，则二级索引的叶子节点占用空间较大。
当然也有适用业务字段直接做主键的：1、只有一个索引；2、该索引必须是唯一索引。这是典型的KV场景。适用业务字段直接做主键可以避免每次查询需要搜索两颗树。

聚簇索引和非聚簇索引

索引和数据存储在一起的叫做聚簇索引，而不与数据存储在一起的叫做非聚簇索引
在InnoDB中，数据必须要和某一个索引列存储在一起，这个索引列可以是主键，如果没有主键，选择第一个没有NULL的唯一键，如果没有唯一键，则会是6个字节的rowid。聚簇索引中B+树的叶子节点存储的数据是整行的记录，而非聚簇索引中B+树的叶子节点中存储的数据仍然是索引结点，只不过是指向对应数据块的指针。

回表

回表我的理解是发生在sql语句中根据非聚簇索引进行查找的时候会发生回表。会先根据非聚簇索引的B+树匹配到对应的叶子节点，查询到对应行记录的聚簇索引的id值，再根据这个id去聚簇索引的B+树去检索整行记录

索引覆盖

 索引的叶子节点中包含了所有查询的所有列，则此时不需要回表，这个过程叫做索引覆盖。再某些场景中，可以考虑将要查询的所有列都变成组合索引，此时会使用索引覆盖，加快查询效率

最左匹配

组合索引要遵循最左匹配原则，在做查询的时候，会按照组合索引从左到右的顺序进行等值查询（sql的优化器会调整sql语句中等值查询语句的顺序），直到遇到不是等值查询，发生索引失效，遇到范围查询（>、<、between、like）就会停止匹配
最左匹配原则就是指在组合索引中，如果你都SQL语句中用到了联合索引中的最左边的索引，那么这条SQL就可以利用这个联合索引去进行匹配

最左匹配原则的原理

索引的底层是一个b+树，组合索引也是这样，假如创建一个(a,b,c)的联合索引，对应的b+树的非叶子节点存储的是第一个关键字的索引a，而叶子接待你存储的是三个关键字的数据，这里可以看出a是有序的，而b,c都是无序的，但是当在a相同的时候，b是有序的，b相同的时候，c又是有序的

在建立联合索引的时候，如何安排索引内的字段顺序

第一原则是，如果通过调整顺序，可以少维护一个索引，那么这个顺序往往就是需要优先考虑采用的。第二原则是考虑空间问题，如果查询条件中既有(a,b)，又有b，那么就要维护(a,b),b两个索引，需要将b设置为占用空间小的字段。

索引下沉

索引下沉可以减少回表的次数，索引下沉只对二级索引有效，在对多个字段进行约束查询时，使用索引下沉后，会先在存储引擎中对第一个字段进行查询，再根据其他字段进行数据筛选，将筛选的结果返回给server层。如果没有索引下沉，在根据第一个字段进行查询后，将查询到的数据返回给server层，在server层进行数据筛选 。

常见的存储引擎

  1. InnoDB，MySQL默认的事务型存储引擎，优点是支持事务和崩溃修复功能，引入了行级锁和外锁约束；缺点是占用的数据空间相对较大；适用的场景是需要事务支持，并且有较高的并发读写频率
  2. MyISAM存储引擎，对于只读数据或者表比较小，可以容忍修复操作，可以适用MyISAM，将表存储在两个文件中；优点是访问速度快，缺点是MyIASM不支持事务和行级锁，不支持事务和行级锁，不支持崩溃后的安全恢复，也不支持外键。适合场景，对事务完整性没有要求，表的数据都会只读的
  3. MEMORY，将数据全部放在内存中，访问速度较快，但是一旦系统崩溃的话，数据都会丢失，默认适用hash索引，将键的hash和指向数据行的指针保持在hash索引中；优点是访问速度较快；缺点是hash索引数据不是按照索引值顺序存储，无法用于排序，不支持部分索引匹配查找，因为hash索引是适用索引列的全部内容来计算hash值的；只支持等值比较，不支持范围查询；当出现hash冲突时，存储引擎需要遍历链表中所有的行指针，逐行进行比较，直到找到符合条件的行
  4. ARCHIVE存储引擎，非常适合存储大量独立的，作为历史记录的数据，提供了压缩功能，拥有高效的插入速度，但是这种引擎不支持索引，所以查询性能较差### 普通索引和唯一索引
    唯一索引的字段值唯一，普通索引和change buffer相结合可以起到提升性能的作用
    redo log 主要节省的是随机写磁盘的 IO 消耗（转成顺序写），而 change buffer 主要节省的则是随机读磁盘的 IO 消耗。

优化器的逻辑

事务

当前读和快照读

事务的四个特性

原子性、持久性、一致性、隔离性
原子性：事务内的操作要么都发生，要么都不发生，通过undo log来实现
持久性：事务一旦提交，它对数据库中的数据的改变就应该是永久的，通过redo log来实现
一致性：事务执行的结果使数据库从一个一致性状态到另一个一致性状态
隔离性：一个事务操作的使用的数据对并发事务是隔离的，通过MVCC来实现

四种隔离级别

innodb中事务的隔离级别分为读未提交，读已提交，可重复读，可串行化这四种

  1. 读未提交：会读到并发事务未提交的数据，会尝试脏读，幻读，不可重复读的问题
  2. 读已提交：会读到并发事务已提交的数据，会产生不可重复读，幻读的问题
  3. 可重复读：可保证同一事务的多个实例在并发读取数据时，能够看到同样的数据，不过存在幻读的问题，幻读的出现是因为事务中既有快照读。
  4. 可串行化：强制事务排序执行，使之不可能会冲突### Mysql的日志有哪些
    innodb事务日志有redo log和undo log
    redo log ：事务开始之前，开始操作数据之前，将要操作的数据放入redo log中，保证了MySQL的持久性，二阶段提交，会先写日志后写数据，这样可以防止在写数据时宕机
    undo log 保存了之前的版本的数据，可以用来回滚行记录到某个版本，保证了数据的原子性；里面的数据可作为数据旧版本快照供其他并发事务进行快照读
    mysql中还有binlog，里面记录了数据库表结构变更和表中数据修改（insert、update、delete）的二进制日志，不会记录对查询的操作
    二阶段提交是指在对表结构进行变更和表中数据进行修改的操作的时候，会先写redo log ，redo log的状态为prepare，再写binlog ，之后提交事务，redo log的状态变为commit

bin log和redo log的区别

  1. bin log会记录所有日志记录，包括InnoDB、MYISAM等存储引擎的日志；redo log只记录innoDB自身的事务日志
  2. bin log只在事务提交之前写入磁盘，一个事务只写一次；而在事务进行过程，会有redo log不断写入磁盘
  3. bin log是逻辑日志，记录的是SQL语句的原始逻辑；redo log是物理日志，记录的是在某个数据页上做了什么修改。binlog只是逻辑日志，只记录原始的更新，删除，插入的逻辑，不记录数据也上具体的修改。因此无法只是用binlog，将redo log抛弃。
  4. redo log是循环写的；binlog是可以追加写入的，追加写是指binlog文件写到一定大小后会切换到下一个，并不会覆盖到以前的日志。

MVCC多版本并发控制

MVCC就是同一份数据保留多版本的一种方式，进而实现并发控制。在查询的时候，通过read view和版本链找到对应版本的数据
在每行记录中有3个隐藏的字段，分别为创建或最后一次修改此条记录的事务id，rowid，回滚指针，其中回滚指针为创建或最后一次修改该记录行的事务id，指向undo log中的原记录。但有一个事务在修改，而其他事务读取时，会读取之前版本的值。当使用事务更新行记录的时候，1、用排查所锁住该行，2、将该行原本的值拷贝到undo log，作为旧版本用于回滚；3、修改当前行的值，生成一个新版本，更新事务id，使回滚指针指向旧版本的记录，这样就形成一条版本链。
MVCC的最大的好处就是读不用加锁，读写不冲突，极大的增加了Mysql的并发性能，保证了隔离性

mysql中的锁

按锁的粒度分，数据库的锁可分为行级锁，表级锁，页级锁。行级锁开销大，加锁慢；会出现死锁；锁的粒度最小，冲突的可能性最小，并发度最高。表级锁开销小，加锁快；不会出现死锁；锁定粒度大，但出现锁冲突的概率最高。页级锁的粒度介于行级锁和表级锁的中间。
按锁的类别上来讲，有共享锁和排它锁。共享锁（lock in share mode）又称读锁，当用户要进行数据的读取时，对数据加上共享锁，共享锁能加上多个，排它锁（for update）又称写锁，当用户要进行数据的写入时，对数据加上排它锁，排它锁对共享锁和其他的排它锁都互斥。
解决mysql中可重复读的幻读问题的方法是间隙锁

MySQL里面表级锁分为表锁和元数据锁

  1. 表锁的语法是 lock tables … read/write。**与 FTWRL 类似，可以用 unlock tables 主动释放锁，也可以在客户端断开的时候自动释放。需要注意，lock tables 语法除了会限制别的线程的读写外，也限定了本线程接下来的操作对象。举个例子, 如果在某个线程 A 中执行 lock tables t1 read, t2 write; 这个语句，则其他线程写 t1、读写 t2 的语句都会被阻塞。同时，线程 A 在执行 unlock tables 之前，也只能执行读 t1、读写 t2 的操作。连写 t1 都不允许，自然也不能访问其他表。一般不使用这种命令来控制并发
  2. 另一种表级锁是元数据锁（meta data lock）MDL，当对一个表做增删改查操作的时候，加MDL读锁，当对表做结构变更操作的时候，加MDL写锁。MDL锁是系统默认会加的。读锁之间不互斥，因此可以有多个线程对同一张表增删改查。读写锁之间，写锁之间是互斥，用来保证变更表结构操作的安全性。因此，如果有两个线程要同时给一个表加字段，其中一个要等另一个执行完之后才能开始执行。 当对表进行字段修改的时候，需要先让读锁结束，需要先解决长事务，事务不提交，就会一致占用MDL锁，在MySQL的information_schema库的innodb_trx表中，可以查到当前执行中的事务，如果要做DDL变更的表刚好有长事务在执行，要考虑先暂停DDL，或者kill掉这个长事务。
  3. 在生产环境中如果对表字段进行修改？但考虑一下这个场景。如果你要变更的表是一个热点表，虽然数据量不大，但是上面的请求很频繁，而你不得不加个字段，你该怎么做呢？这时候 kill 可能未必管用，因为新的请求马上就来了。比较理想的机制是，在 alter table 语句里面设定等待时间，如果在这个指定的等待时间里面能够拿到 MDL 写锁最好，拿不到也不要阻塞后面的业务语句，先放弃。之后开发人员或者 DBA 再通过重试命令重复这个过程。MariaDB 已经合并了 AliSQL 的这个功能，所以这两个开源分支目前都支持 DDL NOWAIT/WAIT n 这个语法。

MySQL的行锁

MySQL的行锁是引擎层有各个引擎自己实现的，并不是所有的引擎都支持行锁，比如MyISAM引擎就不支持行锁。
MySQL的行锁是两阶段锁，也就是说，在InnoDB事务中，行锁是在需要的时候才加上，并不是不需要了就立刻释放，而是要等到事务结束时才释放。因此，如果在一个事务中需要锁多个行，要把最有可能造成锁冲突、最有可能影响并发度的锁尽量往后放。

MySQL中的死锁问题

在并发系统中不同线程出现循环资源依赖，涉及的线程都在等待别的线程释放资源时，就会导致这几个线程都进入无限等待的状态，称为死锁。当出现死锁，有两种策略：一种策略是直接进入等待，直到超时，线程退出，其他线程就可以继续执行。这个超时时间通过参数innodb_lock_wait_timeout来设置。这种策略时间长业务上无法容忍，时间短会误伤。另一种策略是发起死锁检测，发现死锁后，主动回滚死锁链条中的某一个事务，让其他事务得以继续执行，将参数innodb_deadlock_detect设置为on，表示开始死锁检测。死锁检测的过程中会销号掉大量的CPU资源。

分库分表

为什么要分库分表

当表中数据量很大的时候，sql语句执行的效率会变得很低，所需的磁盘容量很大，扛不住高并发。

怎么分表分库

水平拆分，将表按range来分，每个表一段连续的记录
垂直拆分，每个表包含部分字段，将少的访问频率高的字段放在一个表中，将多的访问频率低的字段放在另一个表中
水平拆分的优点是容易扩展，缺点是容易产生热点问题；垂直拆分的缺点是不易扩展

读写分离，主从同步

为什么要做主从同步

在主服务器上执行写入和更新，在从服务器上提供读取功能，可以动态的调整从服务器的数量，从而调整整个数据库的性能。
可以在从服务器上进行数据分析的功能，从而提高主服务器的性能
主从同步还可以进行数据备份的功能

如何实现主从复制

主从复制主要有三个线程：binlog线程、io线程、SQL线程
binlog线程：负责将主服务器上的数据更改写入到二进制日志
io线程：负责从助服务器上读取binlog，并写入到服务器的中继日志中
SQL线程：负责读取中继日志，解析出主服务器中已经执行的数据更改并从服务器中重放
主服务器在每个事务更新数据之前，将该操作写入binlog，从站开启一个io线程，读取binlog，将binlog的内容复制到中继日志中。从站的SQL线程会读取中继日志，顺序执行该日志中的SQL事件，从而和主站的数据保持一致

左连接、右连接、内连接、全外连接的区别

  1. 左向外联接的结果集包括 left join子句中指定的左表的所有行，而不仅仅是联接列所匹配的行。如果左表的某行在右表中没有匹配行，则在相关联的结果集行中右表的所有选择列表列均为空值。
  2. 右连接返回右表中所有记录和左表中连接字段相等的记录
  3. 内连接有两种，显式的和隐式的，返回连接表中符合连接条件和查询条件的数据行。隐式的直接用where子句，不是inner join子句
  4. 完整外部联接返回左表和右表中的所有行。当某行在另一个表中没有匹配行时，则另一个表的选择列表列包含空值。如果表之间有匹配行，则整个结果集行包含基表的数据值。 ### delete、truncate、drop的区别
