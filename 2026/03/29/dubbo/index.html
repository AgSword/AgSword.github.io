dubbo

整体架构

各层说明：

  • config配置层：对外配置接口，以ServiceConfig和ReferenceConfig为中心，可以直接初始化配置类，也可以通过Springboot配置文件进行配置
  • proxy服务代理层：服务接口透明代理，生成服务的客户端和服务端，以Serviceproxy为中心，扩展接口为ProxyFactory
  • registry注册中心层：封装服务地址的注册和发现，以服务URL为中心，扩展接口为RegistryFactory，Registry，RegistryServcie
  • cluster路由层：封装多个提供方的路由和负载均衡，并桥接注册中心，以Invoker为中心，扩展接口为Cluster，Directory，Router，LoadBalance
  • monitor监控层：RPC调用次数和调用时间监控，以Statistics为中心，扩展接口为MonitorFactory，Monitor，MonitorService
  • protocol远程调用层：封装RPC调用，以invocation，Result为中心，扩展接口为Protocl，Invoker，Exporter
  • exchange信息交换层：封装请求响应模式，同步转异步，以Request,Response为中心，扩展接口为 Exchanger , ExchangeChannel , ExchangeClient , ExchangeServer
  • transport网络传输层：实现多种网络### 调用过程
  1. 服务提供方启动，开启Netty服务，创建Zookeeper客户端，向注册中心注册服务
  2. 服务消费方启动，通过Zookeeper向注册中心获取服务提供方列表，与服务提供方通过Netty建立长连接。
  3. 服务消费方通过接口开始远程调用服务，ProxyFactory通过初始化Proxy对象，创建动态代理对象
  4. 通过invoke方法，层层包装生成了Invoker对象
  5. invoker通过路由，负载均衡选择一个最合适的服务提供方，再经过各种过滤器，proxy层将其包装成一个新的DubboInvoker对象
  6. exchange层将DubboInvoker对象包装成一个Request对象，该对象通过序列化经过NettyClient传输倒服务提供方的NettyServer端
  7. 到了服务提供方之后，再通过反序列化、协议解密等操作生成一个DubboExporter对象，再层层传递处理，会生成一个服务提供端Invoker对象
  8. 这个Invoker对象会调用本地服务，获得结果再通过层层回调返回到服务消费方，服务消费方拿到结果后，再解析获得最终结果## Dubbo中的SPI
    Dubbo 中某个接口被 @SPI注解修饰时，就表示该接口是扩展接口****。@SPI 注解的 value 值指定了默认的扩展名称，例如，在通过 Dubbo SPI 加载 Protocol 接口实现时，如果没有明确指定扩展名，则默认会将 @SPI 注解的 value 值作为扩展名，即加载 dubbo 这个扩展名对应的 org.apache.dubbo.rpc.protocol.dubbo.DubboProtocol 这个扩展实现类，相关的 SPI 配置文件在 dubbo-rpc-dubbo 模块中

通过遍历DubboInternalLoadingStrategy ，DubboLoadingStrategy ，ServicesLoadingStrateg三个策略中的directory()加载所有扩展类
在加载扩展类的过程中会实现Dubbo IOC和AOP。Dubbo IOC实现原理是执行实现类的set方法。Dubbo AOP的实现原理是判断是否有类的拷贝构造函数的参数为扩展接口，如果有，则认为此类是扩展实现类的装饰者（即此处用到了装饰者模式：在不改变原类以及不适用继承的情况下，动态地将责任附加在对象上，从而实现动态扩展一个对象的功能，装饰真实的对象），实例化这个包装类代替扩展实现类
有些扩展不想在框架启动阶段被加载，而是希望在扩展方法被调用时根据运行时参数进行加载，即根据参数动态加载实现类。在dubbo中使用Adaptive注解来实现动态加载，此注解可以放在扩展接口上也可以放在扩展接口的函数上
dubbo中的自动激活。org.apache.dubbo.rpc.Filter 接口有非常多的扩展实现类，在一个场景中可能需要某几个 Filter 扩展实现类协同工作，而另一个场景中可能 需要另外几个实现类一起工作。这样，就需要一套配置来指定当前场景中哪些 Filter 实现是可用的，这就是 @Activate 注解要做的事情

服务暴露的过程

Dubbo框架将服务暴露分为两大部分，第一部分时将持有的服务实例通过proxy转换为invoker，第二部分将invoker通过具体的protocol转为exporter。
整个 Provider 节点的**启动入口是 DubboBootstrap.start() 方法。**不仅是直接通过 API 启动 Provider 的方式会使用到 DubboBootstrap，在 Spring 与 Dubbo 集成的时候也是使用 DubboBootstrap 作为服务发布入口。具体逻辑在 DubboBootstrapApplicationListener 这个 Spring Context 监听器中。
