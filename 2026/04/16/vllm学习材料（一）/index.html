vllm学习材料（一）

4P是什么

数据并行DP

**模型全量复制，数据切分。**每张显卡上都有一份完整的模型，训练时，数据被分为N份（N为显卡数）

张量并行TP

**层内切分，张量分割。**针对模型某一层内部的矩阵乘法（如 Y = W X）。权重矩阵 W 太大放不进一张卡，就把 W 竖着切一刀或横着切一刀，放在两张卡上。计算时，两张卡分别算自己那一半，然后再拼接或求和。

流水线并行PP

**层间切分，按阶段接力。**显卡1负责第1-10层，显卡2负责第11-20层…，数据像流水线上的零件，经过卡1算完传给卡2，以此类推。

专家并行EP

**针对MoE架构的特殊切分。**模型中有N个不同的前馈网络（叫专家），每个Token只激活其中Top-K个，EP将不同的专家放置在不同的显卡上。

vLLM的使用

离线批量处理

离线推理是vLLM最基础的使用模式，通过LLM类在Python脚本中批量生成文本，无需启动网络服务。

注意点

  • vLLM 默认从 Hugging Face 下载模型。如需使用 ModelScope，需设置环境变量 export VLLM_USE_MODELSCOPE=True
  • llm.generate() 不会自动应用聊天模板。若使用 Instruct/Chat 模型，应改用 llm.chat() 方法
  • vLLM 默认使用模型仓库中 generation_config.json 推荐的采样参数。若需使用 vLLM 自身默认值，请设置 generation_config=“vllm”

批量推理最佳实践

  • 将所有prompt放入一次调用：vLLM的调度器会自动处理内存约束下的分批，将全部prompt合并为一次generate()调用比多次调用更高效。多次调用会引入不必要的引擎重启和缓存失效。
  • 选择合适的 gpu_memory_utilization：默认值 0.9 在大多数场景下表现良好。如果遇到 OOM，适当降低该值；如果 GPU 显存充足且追求最大吞吐，可谨慎提高至 0.95 左右。
  • 利用 get_default_sampling_params()：不同模型可能有推荐的非默认采样参数（如特定的 temperature 或 top_p），通过 llm.get_default_sampling_params() 获取后在此基础上修改，而非从零构造。
  • Pooler 模型需指定 runner=“pooling”：使用嵌入、分类等池化 API 时，务必在初始化时设置 runner=“pooling”，否则引擎将以生成模式加载模型，调用 embed() 等方法会抛出 ValueError。

启动流程解析

  1. setup_server() 在引擎初始化之前抢先绑定端口——这是一个刻意的设计，用于避免与 Ray 的竞态条件。同时它会验证工具调用解析器和推理输出解析器的合法性。
  2. build_async_engine_client() 创建 V1 引擎客户端。在默认的多进程前端模式下，AsyncLLM 会在独立进程中启动引擎核心（EngineCoreProc），API 服务器进程通过 ZMQ 与之通信。
  3. build_app() 构建 FastAPI 应用实例并注册全部路由和中间件，init_app_state() 将引擎客户端与各 Serving 层对象注入应用状态，最后 serve_http() 启动 Uvicorn 事件循环。
